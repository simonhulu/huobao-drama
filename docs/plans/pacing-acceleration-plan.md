# 分镜/解说节奏提速方案（x2）

## 当前问题诊断

以 `episode_id=71` 为例，目前 24 个镜头覆盖了原文大约前 1/2 的情节。拆分结果里出现大量“等待失控”“怔住的沉默”“骤然转冷”“跪地的残影”“心中的海啸，早已平息”这类**纯情绪反应镜头**。它们把同一段内心转折切成了 2-3 个镜头，导致：

- **情绪推进被稀释**：观众刚感受到一个情绪落点，下一个镜头又在重复“他很震惊/我很平静”。
- **情节推进变慢**：真正的动作/决定（举杯、递礼物、离婚协议书）被淹没在反应镜头里。
- **总时长膨胀**：每个镜头即使只配 1-2 句解说，24 个镜头也会把一段年会戏拉到 2-3 分钟。

根本原因：当前 `storyboard_breaker` 和 `storyboard_splitter` 默认以“故事保真”优先，默认保留所有内心、背景、因果、动机；`narrator` 虽然限制“每镜头 1-2 句”，但没有限制“同一个情绪转折点只能给 1 个镜头”。

## 目标

- 以**情绪和情节推进**为唯一准绳，删除或合并重复的情绪反应。
- 内心独白**不再细分**，只保留一个最有力量的落点。
- 整体节奏按 **x2 提速**来设计：同一段戏的镜头数减半，单镜头时长由音频驱动但避免硬撑。
- 用户可以在 UI 上选择“叙事节奏”，重新生成整集。

## 方案（不改动业务表结构的最小可行版）

### 1. 新增 `episodes.pacing_mode` 字段

- 枚举：`standard`（标准，当前效果） / `tight`（紧凑，x2 提速，默认） / `extreme`（极速，单集 8-12 镜）。
- 前端在“分镜”页顶部加一个下拉选择：叙事节奏 · 标准 / 紧凑 / 极速。
- 切分节奏时，后端把该字段写入 `storyboards` 生成提示词的上下文，并触发清空旧分镜/旁白/音频/成片，重新跑 breaker + narrator。

### 2. 修改 `storyboard_breaker` 的 beat 筛选逻辑

当前 SKILL.md 里已经有“AI 漫剧模式”的 beat 筛选，但默认未启用。我们把它升级为按 `pacing_mode` 生效的默认规则：

#### `tight` / `extreme` 模式下强制规则

1. **只保留两类 story beat**：
   - 推进主线的关键事件（动作、决定、信息揭露、冲突升级、反转）。
   - 承载强情绪的核心转折（讽刺、冷漠、爆发、释然）。
2. **合并/删除以下镜头**：
   - 重复的情绪反应（震惊→怔住→沉默→眸色转冷，只保留最有力的一镜）。
   - 纯内心自白细分（“我心里早已平息的海啸”单独一镜 → 合并进“不是装的”这一镜的眼神/台词里）。
   - 低信息过渡（“他等我发作，可我纹丝不动”可以只作为 1 个短镜，不再拆成“他在等 / 我没动”两镜）。
   - 回忆闪回除非直接触发当下决定，否则压缩为 1 个叠化镜头或删掉。
3. **镜头密度上限**：
   - `tight`：每段核心戏 4-6 镜，单集目标 12-18 镜。
   - `extreme`：每段核心戏 2-4 镜，单集目标 8-12 镜。
4. **每个镜头必须有“推进功能”**：在 `description` 第一行写明它是“推进事件”还是“情绪落点”；写不出的镜头删除。

### 3. 修改 `storyboard_splitter` 的拆分策略

当前 splitter 按 `narration + dialogue > 60 字` 或“一个镜头包含多个情绪转折”就拆分，导致内心戏被切太碎。

#### 新规则

- **不把“内心独白的多层递进”当作拆分理由**。一个情绪转折只拆 1 镜。
- **只拆真正多阶段的可见动作/对话**：
  - 动作链：拿起 → 查看 → 放下（最多拆成 2 镜，不要 3 镜）。
  - 对话回合：一问一答可以拆，但不要把一句长台词拆成多个反应镜头。
  - 空间转移：进门 + 走近 + 坐下，最多 2 镜。
- `tight` 模式下阈值提高到 **90 字**，`extreme` 提高到 **120 字**（只在确实信息超载时才拆）。
- 拆分后的子镜头必须各有独立的情节/情绪功能，不能只是“他的反应 / 她的反应”这样成对拆分。

### 4. 修改 `narrator` 的旁白撰写规则

当前规则“每镜头 1-2 句、给对白让位、不扩写”已经不错，但需要再加紧箍咒：

#### `tight` / `extreme` 模式新增硬规则

1. **每镜头最多 1 句解说，极端情况下最多 2 句**。`extreme` 模式强制 1 句。
2. **禁止内心独白细分**：
   - 不允许把“我跪过、求过、把自尊碾碎过——但那都是过去的事了”这种自省拆成多个镜头逐句解说。
   - 只能保留一个情绪结论，例如：“那些跪地求他的日子，过去了。”
3. **只讲“此刻发生了什么”和“情绪落差”**：
   - 不解释背景、不铺陈心理过程。
   - 能用画面（眼神、动作、物件）表达的信息，解说一个字都不说。
4. **给对白让位升级**：
   - 有原声对白的镜头，解说只做情绪标点（1 句），严禁复述、改写、提前剧透对白。
   - 如果该镜头对白已经把意思说完，解说直接留空。
5. **删除过渡性旁白**：
   - “他显然一怔，随即嘴角勾起惯有懒散弧度”这种描述性旁白，交给 `action/result` 和视频画面， narrator 不写。

### 5. 合成阶段避免拖长

当前 `ffmpeg-compose.ts` 已经是“图文叙事镜头时长由 narration + dialogue 实际音频长度决定”，这是对的。但可以做两个微调：

- 在音频结尾后只保留 **0.3-0.5 秒** 留白（而不是默认 1 秒以上），让情绪更紧凑。
- `extreme` 模式下，如果单镜头音频超过 8 秒，提示用户该镜头信息过载，建议拆分或精简旁白。

### 6. UI 与重新生成流程

在分镜页顶部增加：

```
叙事节奏 [ 标准 | 紧凑(x2) | 极速 ]
```

切换时：

1. 弹确认框：会清空当前分镜、旁白、配音、成片并重新生成。
2. 后端清空 `storyboards` / `narration_audio_url` / `composed_video_url` / `merged_url` 等。
3. 按新模式重新跑 `storyboard_breaker` → `storyboard_splitter` → `narrator`。
4. 生成后显示镜头数、预估时长、被合并/删除的镜头摘要。

### 7. 校验与兜底

- 生成后如果镜头数超过“原文有效 beat 数 × 1.5”，自动标红提示“节奏可能仍偏慢，建议检查”。
- 保留每个镜头的 `description` 中的“推进功能”字段，方便人工快速判断是否有拖沓镜头。

## 预期效果

以 `episode_id=71` 为例，按 `tight` 模式重新生成后，预计从 24 镜压缩到 12-14 镜：

| 当前部分镜头 | 合并/删除后 |
|---|---|
| 5-7 镜：纽约试探 / 平静点头 / 遗愿的借口 | 2 镜：他问，我答，遗愿借口被一句话堵死 |
| 8-9 镜：怔住的沉默 / 不再打扰 | 1 镜：他噎住，我转身 |
| 12-14 镜：同情的目光 / 等待失控 / 递礼盒 | 2 镜：我无视他的等待，递出礼盒 |
| 17-18 镜：被推开的C位 / 漠然旁观 | 1 镜（回忆叠化）：上次我被推开，他视而不见 |
| 22-24 镜：跪地的残影 / 不是装的 / 礼盒变沉 | 2 镜：我抬起眼说“不是装的” / 他手中的礼盒突然变沉 |

这样情绪从“讽刺→平静反击→当众羞辱→决绝礼物→回忆一击→最终释然”呈阶梯推进，而不是在每个台阶上反复停留。

## 需要改动的主要文件

1. `backend/src/db/schema.ts`：新增 `episodes.pacing_mode`。
2. `backend/src/db/migrations/`：新增迁移脚本（如果项目用 drizzle migrate）。
3. `backend/src/routes/episodes.ts`：增加 `pacing_mode` 更新接口 + 切节奏时清空并触发重新生成。
4. `backend/src/agents/index.ts` 或 narrator 的默认 prompt：按节奏模式注入不同规则。
5. `skills/narrator/SKILL.md`：更新旁白规则。
6. `skills/storyboard_breaker/SKILL.md`：更新 beat 筛选和镜头密度规则。
7. `skills/storyboard_splitter/SKILL.md`：更新拆分阈值和禁止内心细分规则。
8. `frontend/app/pages/drama/[id]/episode/[episodeNumber].vue`：增加节奏选择器 + 重新生成按钮。

## 风险

- 过度压缩可能丢失原文细腻心理，需要用户审阅合并后的镜头。
- `extreme` 模式可能让单镜头信息过载，建议只对强情节段落使用。
- 修改后需要重新生成图片/音频/成片，耗时较长。

## 建议的下一步

1. 先只做 `tight` 模式（默认），不动 `standard` 的逻辑，保证可回退。
2. 我先用 `episode_id=71` 做一轮测试生成，给你看前后对比。
3. 你确认效果后，再把 `tight` 设为默认并全量重跑。
